INTRODUCCIÓN   El vertiginoso desarrollo de las arquitecturas de software modernas ha impulsado la transición desde sistemas monolíticos tradicionales hacia ecosistemas basados en servicios distribuidos. En estos entornos, la comunicación eficiente entre el cliente frontend y el backend servidor representa el núcleo operativo, requiriendo mecanismos avanzados como WebSockets para superar las limitaciones de latencia inherentes al protocolo HTTP convencional. La sincronización en tiempo real se vuelve un requisito crítico para mantener informados a los usuarios sobre cambios de estado sin necesidad de recargas constantes de pantalla, promoviendo una experiencia fluida e interactiva (Tanenbaum y Van Steen, 2017).   No obstante, la descentralización introduce desafíos complejos relacionados con la concurrencia y la consistencia transaccional cuando múltiples nodos o hilos acceden simultáneamente a los mismos recursos compartidos. El manejo inadecuado de solicitudes concurrentes puede derivar en condiciones de carrera, duplicidad de datos o estados inconsistentes que degradan severamente la confiabilidad del sistema. Ante este panorama, el uso de pools de hilos gestionados de forma asíncrona permite que el servidor delegue tareas pesadas de procesamiento a subprocesos en segundo plano, evitando el bloqueo del hilo de ejecución principal y manteniendo la reactividad general (Coulouris et al., 2011).   Para estructurar de manera robusta estas soluciones, resulta indispensable aplicar metodologías de diseño limpio guiadas por los principios SOLID, los cuales garantizan un acoplamiento débil y una alta cohesión en el código fuente. La separación estricta de responsabilidades entre controladores, servicios de lógica de negocio y capas de acceso a datos facilita no solo la tolerancia a fallos y el manejo controlado de errores, sino también la escalabilidad horizontal del sistema. Comprender estas interacciones a la luz de los fundamentos teóricos distribuidos dota al diseñador de las herramientas necesarias para construir aplicaciones resilientes frente a fallas de conectividad (Martin, 2018).   2. Para esta sección se recomienda el siguiente formato (se recomiendan entre 3 y 5 temas relevantes):   3.1 Consistencia Transaccional en Sistemas Distribuidos   La consistencia transaccional en entornos distribuidos define el grado de coherencia y visibilidad que poseen los datos de forma simultánea a través de los diferentes componentes o nodos del sistema. A diferencia de los entornos locales regidos estrictamente por propiedades ACID, los sistemas distribuidos deben lidiar con latencias de red y procesamientos paralelos que introducen el riesgo de colisiones de datos y lecturas sucias. Para mitigar estos problemas, se implementan técnicas de aislamiento y bloqueos lógicos que aseguran que una transacción incompleta no propague estados erróneos hacia las interfaces de los usuarios, preservando la validez transaccional (Kleppmann, 2017)   3.2 El Teorema CAP y su Impacto Arquitectural   El Teorema CAP, formulado por Eric Brewer, estipula que es matemáticamente imposible para un sistema de datos distribuido garantizar simultáneamente más de dos de las siguientes tres propiedades: Consistencia, Disponibilidad y Tolerancia a particiones. Dado que las redes físicas son inherentemente propensas a sufrir retrasos o pérdidas de paquetes (particiones), los arquitectos de software se ven forzados a elegir entre priorizar la consistencia de los datos o mantener el sistema completamente disponible. En aplicaciones de naturaleza crítica o transacciones comerciales, la consistencia toma prioridad absoluta, sacrificando la disponibilidad inmediata mediante bloqueos temporales de interfaz hasta que el servidor confirme los cambios (Gilbert y Lynch, 2002).   3.3 Protocolo WebSocket y Comunicación Bidireccional   El protocolo WebSocket proporciona un canal de comunicación persistente, full-duplex y de baja latencia sobre una única conexión TCP, transformando la interacción tradicional basada en el modelo de petición-respuesta de HTTP. Al mantener un socket abierto entre el navegador del cliente y el servidor, se elimina la sobrecarga de encabezados repetitivos y se habilita el envío asíncrono de eventos dirigidos desde el backend en el momento exacto en que ocurren. Esta tecnología es fundamental para la construcción de tableros de control interactivos, chats de soporte técnico y sistemas de mensajería asíncrona distribuidos que demandan actualizaciones inmediatas (Fette y Melnikov, 2011).   3.4 Concurrencia y Multithreading Orientado a Servicios   La concurrencia representa la capacidad de un sistema informático para ejecutar múltiples tareas o flujos de instrucciones en intervalos de tiempo superpuestos, maximizando el aprovechamiento del hardware multihilo. En plataformas de servicios web como Spring Boot, cada petición entrante es asignada inicialmente a un hilo de un pool de conexiones HTTP; sin embargo, procesos intensivos o de larga duración deben ser delegados asíncronamente mediante anotaciones especializadas a hilos de trabajo independientes. Esta delegación asegura que el servidor continúe despachando nuevas solicitudes de manera reactiva mientras los subprocesos secundarios completan las operaciones lógicas pesadas (Goetz et al., 2006).   3.5 Principios de Diseño SOLID en Arquitecturas Limpias   Los principios SOLID constituyen un conjunto de cinco directrices de diseño orientado a objetos orientadas a fomentar la legibilidad, mantenibilidad y flexibilidad del código fuente ante cambios requeridos por el negocio. Al aplicar el Principio de Responsabilidad Única (SRP), cada clase o componente dentro de la arquitectura distribuida se restringe a realizar una única función lógica bien definida, separando los controladores de API de los servicios de negocio. Esta separación sistemática reduce drásticamente los efectos colaterales durante las fases de refactorización y permite inyectar dependencias abstractas que facilitan las pruebas de integración distribuidas (Martin, 2018).   4. Por ejemplo, al momento de enviar una orden, si la comunicación sufre un retraso o latencia inusual, el estado se marca localmente en una fase transitoria de espera supervisada por el backend. El sistema no permite dar por completado un pedido de forma autónoma en el cliente hasta que el hilo de procesamiento correspondiente en Spring Boot haya persistido el estado atómico en la base de datos relacional. De este modo, la disponibilidad global se sacrifica momentáneamente para blindar la coherencia matemática de las transacciones operativas.   4.7 APLICACIÓN DE CONCEPTOS PREVIAMENTE REVISADOS   ConceptoAplicación en la soluciónCliente-servidorImplementado mediante el consumo explícito de controladores @RestController de Spring Boot desde componentes React a través de peticiones asíncronas con la API fetch de JavaScript. La incorporación de bloques sincronizados en el manejador del socket impidió la superposición de datos de red, cumpliendo con las condiciones de exclusión mutua necesarias para entornos de memoria compartida (Goetz et al., 2006).   Por otra parte, la decisión arquitectónica de sacrificar la disponibilidad instantánea en favor de la consistencia rigurosa frente a particiones teóricas se alinea armónicamente con las soluciones industriales analizadas por Kleppmann, en donde se demuestra que es preferible forzar un estado local intermedio editable bajo supervisión centralizada que permitir transacciones locales descoordinadas que corrompan el balance operacional del negocio (Kleppmann, 2017).   7. CONCLUSIONES   Se logró implementar con éxito un esquema de consistencia transaccional robusto en el sistema distribuido "Bar Distribuido", logrando aislar las operaciones simultáneas de compra mediante el uso de pools de hilos de ejecución asíncronos y asilados.   La comunicación interactiva y bidireccional en tiempo real se consolidó de manera eficiente mediante la integración del protocolo WebSocket, posibilitando una sincronización inmediata de eventos sin saturar la red con técnicas ineficientes de sondeo repetitivo.   El diseño e implementación inicial demostró la aplicabilidad práctica de los principios SOLID y el Teorema CAP, ratificando la necesidad de dar prioridad a la consistencia informática sobre la disponibilidad ante escenarios de fallos distributivos de red.   8.
